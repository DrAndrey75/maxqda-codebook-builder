# Contributing

Спасибо за интерес к проекту! Ниже — минимальные правила и процесс, чтобы изменения было проще принять и поддерживать.

## Что можно улучшать
- Очистка/нормализация кодбука и генерация вариантов терминов.
- Извлечение и фильтрация аббревиатур.
- Качество exact‑match по корпусу (скорость/точность/пороговые значения).
- Экспорт в формат, удобный для импорта в MAXQDA.
- Документация и UX батников/CLI.

## Сообщить об ошибке (Bug report)
1) Проверьте, что проблема воспроизводится на актуальной версии кода.
2) Создайте Issue через шаблон “Bug report” и приложите:
   - команду/шаг пайплайна, который падает,
   - полный текст ошибки/стектрейса,
   - ваш OS и версию Python (`python --version`),
   - входные данные (или минимальный пример без больших PDF).

## Запрос функциональности (Feature request)
Создайте Issue через шаблон “Feature request” и опишите:
- какую задачу вы решаете,
- что должно измениться в результате,
- возможные альтернативы/ограничения.

## Локальная разработка (Windows)
1) Подготовьте окружение:
   - `setup_env.bat`
2) Запуск полного пайплайна:
   - `run_pipeline.bat`
3) Быстрый запуск только финального шага (если кодбук уже подготовлен):
   - `libs\\venv\\Scripts\\python.exe build_codebook.py --codebook-xlsx codebook/common.cleaned.with_context_abbrev.xlsx --input primery --output output/maxqda_codebook.xlsx --max-words 4 --min-frequency 2`

## Стиль и качество кода
- Python: придерживайтесь PEP 8 и существующего стиля проекта (аннотации типов приветствуются).
- Изменения должны быть минимальными и сфокусированными на одной задаче.
- CLI‑интерфейсы скриптов считаем публичными: не ломайте аргументы без веской причины.
- Вывод в консоль: избегайте emoji и экзотических символов (на Windows часто используется cp1251).

## Данные и большие файлы
- Не добавляйте в PR большие PDF‑корпуса или приватные данные.
- Для воспроизведения багов используйте минимальные примеры (1–2 небольших файла).

## Как оформлять Pull Request
1) Форк → ветка → PR.
2) В описании PR:
   - что и зачем изменилось,
   - как это протестировать,
   - какие файлы/шаги пайплайна затронуты.
3) Чек‑лист перед PR:
   - пайплайн запускается (или хотя бы затронутые шаги),
   - документация обновлена, если поведение/CLI изменилось,
   - новые зависимости добавлены в `requirements.txt` (если нужно).
